# Структура локализации

## Структура папок

```
app/i18n/
├── index.ts              # Главный файл с экспортом переводов
├── types.ts              # TypeScript типы для переводов
├── locales/
│   ├── kg/               # Кыргызский язык
│   │   ├── landing.json  # Переводы для главной страницы
│   │   └── layout.json   # Переводы для layout (метаданные)
│   └── ru/               # Русский язык
│       ├── landing.json  # Переводы для главной страницы
│       └── layout.json   # Переводы для layout (метаданные)
└── README.md             # Этот файл
```

## Использование

### Импорт переводов

```typescript
import { getTranslations, Locale } from '@/app/i18n';

// Получить переводы для конкретной страницы
const t = getTranslations('ru', 'landing');

// Использование
console.log(t.header.appName); // "Bite Lens"
console.log(t.hero.title.part1); // "Сфотографируй еду,"
```

### Добавление нового языка

1. Создайте папку с кодом языка в `locales/` (например, `en/`)
2. Создайте JSON файлы для каждой страницы
3. Добавьте язык в `index.ts`:
   - В тип `Locale`
   - В массив `locales`
   - В объект `translations`
   - В объект `localeNames`

### Добавление новой страницы

1. Создайте JSON файл в каждой языковой папке (например, `dashboard.json`)
2. Добавьте импорты в `index.ts`
3. Добавьте в объект `translations`

## Структура JSON файлов

Каждый JSON файл должен иметь логическую структуру, группирующую связанные переводы:

```json
{
  "section": {
    "subsection": {
      "key": "значение"
    }
  }
}
```

### Пример для страницы

```json
{
  "header": {
    "title": "Заголовок",
    "nav": {
      "home": "Главная",
      "about": "О нас"
    }
  },
  "content": {
    "description": "Описание"
  }
}
```

## Поддерживаемые языки

- **kg** - Кыргызский (Кыргызча)
- **ru** - Русский (по умолчанию)

## Рекомендации

1. Используйте понятные ключи на английском языке
2. Группируйте переводы по секциям страницы
3. Сохраняйте одинаковую структуру во всех языках
4. Для массивов используйте объекты с индексами или именованные ключи
5. Проверяйте валидность JSON перед коммитом
